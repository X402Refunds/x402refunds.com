## Step 2: What Sellers Need to Do (Signed Evidence)

### The Technical Implementation

**It's actually quite simple** - sellers sign ONE thing and send TWO headers:

```javascript
// STEP 1: Build the canonical payload (EXACT format)
const payload = JSON.stringify({
  request: {
    method: "POST",
    path: "/v1/chat/completions",
    headers: {...},
    body: {...}
  },
  response: {
    status: 500,  // Or 200, whatever they returned
    body: "Internal Server Error"  // Their actual response
  },
  amountUsd: 0.05,
  x402paymentDetails: {
    currency: "USDC",
    blockchain: "base",
    transactionHash: "0x123...",
    fromAddress: "0xBuyer...",
    toAddress: "0xSeller...",
    timestamp: "2025-11-07T15:30:00Z"  // Optional
  }
});

// STEP 2: Sign it with Ed25519 (standard crypto library)
const signature = ed25519.sign(payload, privateKey);

// STEP 3: Return BOTH payload and signature in headers
response.headers.set('X-Payload', base64Encode(payload));      // ← The exact string that was signed
response.headers.set('X-Signature', base64Encode(signature));  // ← The signature proving it
response.headers.set('X-Dispute-URL', disputeUrl);
```

**Why send the payload?** Prevents JSON formatting mismatches. Buyer forwards the EXACT string seller signed - no reconstruction needed, tamper-proof.

## Step 3: Canonical Payload Format

**Sellers must use this EXACT format** when building the payload to sign:

```javascript
const payload = JSON.stringify({
  request: {
    method: string,        // "POST", "GET", etc.
    path: string,          // "/v1/chat/completions"
    headers: object,       // Request headers (optional)
    body: object           // Request body (optional)
  },
  response: {
    status: number,        // 200, 500, 503, etc.
    body: string           // Response body as string
  },
  amountUsd: number,       // Transaction amount in USD
  x402paymentDetails: {
    // REQUIRED FIELDS:
    currency: string,           // "USDC", "ETH", "SOL", etc.
    blockchain: string,         // "base", "ethereum", "solana", etc.
    transactionHash: string,    // Blockchain tx hash (proof of payment)
    fromAddress: string,        // Buyer's wallet address
    toAddress: string,          // Seller's wallet address
    
    // OPTIONAL FIELDS:
    timestamp: string,          // ISO 8601 format
    blockNumber: number,        // Block number
    contractAddress: string,    // Token contract address
    layer: string,              // "L1" or "L2"
    explorerUrl: string         // Blockchain explorer URL
  }
});

// Use JSON.stringify with NO formatting (compact, no spaces)
// Field order MUST match above for deterministic signatures
```

**Why this matters:** Buyer will forward your EXACT payload string to Consulate. Any formatting differences will break signature verification.

## Step 4: Buyer Agent Auto-Files Disputes

**Buyer agent code** (ONE tool call):

```javascript
// After receiving seller's response
const response = await fetch(sellerApiUrl, {...});

// If bad output detected
if (response.status >= 400 || detectBadOutput(response.body)) {
  
  // File dispute - Copy 3 headers from seller, that's it!
  await consulate_file_dispute({
    plaintiff: "wallet:0xBuyer123...",  // Buyer's wallet address
    disputeUrl: response.headers.get('X-Dispute-URL'),
    description: "API returned 500 error but charged me",
    evidencePayload: response.headers.get('X-Payload'),      // ← Exact signed string
    signature: response.headers.get('X-Signature')           // ← Signature proving it
  });
}
```

**That's it!** Buyer just copies 3 header values. No reconstruction, no tampering possible.
